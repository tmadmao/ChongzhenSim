# 崇祯皇帝模拟器 - 架构文档

## 概述

崇祯皇帝模拟器是一个基于React + TypeScript的历史策略模拟游戏，让玩家扮演明朝崇祯皇帝，在风雨飘摇的明末时局中力挽狂澜。项目采用现代化的前端架构，结合轻量级后端服务，实现复杂的国家管理和AI增强的剧情体验。

## 核心目标

- **历史还原**：准确模拟明末崇祯年间的政治、经济、军事格局
- **沉浸体验**：提供皇帝角色的深度角色扮演
- **策略深度**：实现复杂的决策系统和长期后果
- **AI增强**：利用大语言模型生成动态剧情和智能对话
- **技术先进**：采用现代Web技术栈确保性能和可维护性

## 技术栈

| 层级 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 前端框架 | React | 19.x | UI组件和状态管理 |

## 架构设计原则

### 1. 分层架构
项目采用清晰的分层设计，确保职责分离和可维护性：

```
┌─────────────────┐
│   表现层 (UI)   │  ← React组件，Tailwind样式
├─────────────────┤
│   业务逻辑层    │  ← Systems、Engines、Stores
├─────────────────┤
│   数据访问层    │  ← Database、Queries、APIs
├─────────────────┤
│   基础设施层    │  ← Utils、Config、Types
└─────────────────┘
```

### 2. 模块化设计
- **组件化**：每个UI模块独立封装
- **服务化**：业务逻辑抽象为独立服务
- **插件化**：功能模块可插拔扩展

### 3. 状态管理策略
- **单一数据源**：游戏状态集中管理
- **不可变更新**：使用深拷贝确保状态纯净
- **事件驱动**：通过事件总线解耦组件通信

## 前端架构

### 组件架构

```
src/components/
├── layout/          # 布局组件 (Layout, BottomBar, StatusBar)
├── panels/          # 功能面板 (CourtPanel, MilitaryPanel等)
├── map/            # 地图相关 (GameMap, ProvinceInfoPanel)
├── event/          # 事件系统 (EventPanel, ScenarioEventPanel)
├── finance/        # 财政面板 (FinancePanel)
├── minister/       # 大臣系统 (MinisterChatPanel)
├── decree/         # 诏书系统 (DecreePanel)
├── settings/       # 设置面板 (SettingsPanel)
└── debug/          # 调试工具 (DebugPanel)
```

**组件设计原则**：
- **懒加载**：使用React.lazy实现代码分割
- **错误边界**：每个主要组件包装ErrorBoundary
- **性能优化**：使用React.memo和useMemo避免不必要重渲染

### 状态管理架构

```
src/store/
├── gameStore.ts      # 游戏主状态 (游戏循环、回合管理)
├── provinceStore.ts  # 省份状态 (省份数据、税收管理)
├── financeStore.ts   # 财政状态 (收支记录、预算管理)
├── courtStore.ts     # 朝廷状态 (大臣管理、廷议系统)
├── policyStore.ts    # 政策状态 (国策树、政策效果)
├── navigationStore.ts # 导航状态 (面板切换、UI状态)
├── settingsStore.ts  # 设置状态 (游戏配置、主题)
└── themeStore.ts     # 主题状态 (明朝风格配色方案)
```

**Zustand状态管理特点**：
- **轻量级**：无Redux的复杂性，API简洁
- **持久化**：自动本地存储游戏进度
- **中间件**：支持日志、调试和持久化
- **类型安全**：完整的TypeScript类型定义

## 后端架构

### API设计

```
server/
└── index.ts          # Express服务器主文件
```

**API端点**：
- `GET /api/health` - 健康检查
- `POST /api/minister-chat` - 大臣对话（流式响应）

**设计特点**：
- **轻量化**：仅处理AI对话，减少复杂性
- **代理模式**：前端不直接调用AI API，提高安全性
- **流式响应**：支持实时对话体验

### AI集成架构

```
src/api/
├── llmClient.ts      # LLM客户端 (多模型支持、错误处理)
├── eventContext.ts   # 事件上下文构建 (游戏状态转换)
├── index.ts          # API入口 (统一导出)
└── schemas.ts        # 数据验证 (Zod模式)
```

**AI集成策略**：
- **多模型支持**：OpenAI GPT-4、DeepSeek等
- **上下文管理**：动态构建游戏状态上下文
- **错误降级**：AI失败时提供本地fallback
- **成本控制**：Token使用统计和限制

## 数据层架构

### 数据库设计

```
src/db/
├── database.ts       # 数据库初始化和核心操作
└── queries/          # 查询构建器和优化
```

**数据存储策略**：
- **客户端优先**：使用SQL.js实现离线存储
- **关系型设计**：完整的表结构和外键关系
- **版本管理**：数据库schema版本控制

### 数据模型

```sql
-- 核心数据表
provinces              -- 省份信息
treasury_transactions  -- 财政流水
game_snapshots         -- 游戏存档
minister_logs          -- 大臣行动记录
```

**数据流设计**：

3. **同步**：前端状态与数据库保持同步

## 核心系统架构

### 游戏循环系统

```
src/core/
├── gameLoop.ts        # 游戏主循环 (回合结算逻辑)
├── eventBus.ts        # 事件总线 (组件间通信)
├── types.ts           # 类型定义 (完整的类型系统)
└── fieldKeys.ts       # 字段常量 (避免魔法字符串)
```

**游戏循环流程**：
1. **应用变更**：处理ChangeQueue中的累积变更
2. **税收结算**：计算各省份税收收入
3. **财政结算**：处理国家收支和预算
4. **省份更新**：更新省份状态和发展
5. **大臣更新**：处理大臣忠诚度和行动
6. **国策研究**：推进国家政策研究
7. **事件处理**：触发剧本事件
8. **日期推进**：进入下一回合
9. **状态保存**：持久化游戏状态

### 业务系统

```
src/systems/
├── taxSystem.ts       # 税收系统 (税收计算逻辑)
├── financeSystem.ts   # 财政系统 (预算和支出管理)
├── courtSystem.ts     # 朝廷系统 (大臣管理和廷议)
├── scenarioEngine.ts  # 剧本引擎 (事件和剧情管理)
└── policySystem.ts    # 政策系统 (国策树管理)
```

**系统设计原则**：
- **单一职责**：每个系统专注特定业务领域
- **依赖注入**：通过构造函数注入依赖
- **接口抽象**：定义清晰的系统接口
- **测试友好**：系统逻辑独立，便于单元测试

### 引擎层

```
src/engine/
├── AccountingSystem.ts # 会计引擎 (精确的财务计算)
├── ChangeQueue.ts      # 变更队列 (状态变更管理)
└── [其他引擎]          # 扩展引擎
```

**引擎设计特点**：
- **高性能**：优化计算逻辑，减少重复计算
- **事务性**：确保状态变更的原子性
- **可扩展**：支持新引擎的动态加载

## 架构优化建议

### 1. 性能优化

**前端性能**：
- **代码分割**：按路由和功能模块分割
- **懒加载**：组件和数据的延迟加载
- **虚拟化**：长列表使用虚拟滚动
- **缓存策略**：API响应和计算结果缓存

**数据库性能**：
- **索引优化**：关键查询字段建立索引
- **查询优化**：避免N+1查询，使用JOIN
- **批量操作**：减少数据库往返次数
- **内存缓存**：热点数据缓存到内存

### 2. 可维护性提升

**代码质量**：
- **严格模式**：启用TypeScript严格模式
- **测试覆盖**：核心逻辑单元测试覆盖80%+
- **代码审查**：建立代码审查流程
- **文档完善**：API和架构文档同步更新

**架构重构**：
- **服务层**：提取业务逻辑到独立服务
- **中间件**：添加日志、监控、错误处理中间件
- **配置管理**：环境配置和功能开关
- **依赖管理**：清理未使用依赖，升级关键依赖

### 3. 可扩展性设计

**模块化扩展**：
- **插件系统**：支持第三方功能插件
- **主题系统**：可配置的视觉主题
- **本地化**：多语言支持框架
- **模组支持**：用户自定义剧本和规则

**API设计**：
- **版本控制**：API版本管理和兼容性
- **文档生成**：自动生成OpenAPI文档
- **类型安全**：前端后端共享类型定义
- **监控告警**：API性能和错误监控

### 4. 安全性增强

**前端安全**：
- **输入验证**：所有用户输入验证和清理
- **XSS防护**：内容安全策略和转义
- **CSRF防护**：请求验证和token机制

**后端安全**：
- **认证授权**：用户身份验证和权限控制
- **API限流**：防止滥用和DDoS攻击
